> **날짜**: 2026.03.24  
> **참여**: 연구자 (한의학 임상) + Claude  
> **목적**: 한의학 처방 벡터 분석 시스템 기획 및 스타터 킷 구축

## 1. 연구자의 초기 구상

### 1.1 한의학 처방 체계에 대한 관점

연구자는 한의학의 처방이 연역적으로 발전했다기보다는, **특정 처방 데이터를 귀납적으로 설명할 수 있는 동양철학 모델을 바탕으로 뼈대를 잡고, 이후 사변적인 내용은 임상 현장에서 제거해왔다**고 보았다.

따라서:
- 동양철학에 기반한 의론을 세우기보다는
- 기존부터 효능을 인정받아 전승되어 내려온 처방의 효과를 인정한다는 전제하에
- 각 본초 또는 혼합물의 **약리학적, 생화학적 근거**를 활용하여 현대적으로 임상 현장에 응용하는 것이 필요하다고 주장

### 1.2 증상-처방 대응 관계 프레임워크

의학사적으로 어떠한 증상이나 특정 변증(한의학에서의 증후군)은 그것을 치료하는 처방과 대응된다는 관점:

1. **기존 처방 매칭**: 환자의 증상 조합이 특정 병증에 가장 부합하는 처방을 사용
2. **새로운 처방 조합**: 각 증상을 가장 잘 대표하는 처방이나 핵심 본초의 조합을 방제학적 원리에 벗어나지 않는 한에서 운용

### 1.3 에이전트 요구사항

- Golden standard 처방군 선정 (방약합편, 동의보감, 상한론 등)
- 본초학, 생화학적, 약리학적 지식 추가
- **기능 목표**:
  - (1) 특정 증상을 치료하는데 효능이 좋은 처방 추천
  - (2) 특정 처방에 대한 이해/인수분해
  - (3) 구전 처방이나 다수 후보 처방 분석
- **핵심 제약**: 기존 LLM 데이터 최대한 배제, 연구자가 입력한 데이터 안에서만 사고 (환각 방지)

## 2. Claude의 평가

### 2.a 동양의학사적 관점에서의 평가

**타당한 부분:**
- 상한론이 대표적 사례. 장중경은 음양오행 이론을 거의 언급하지 않고, 임상 관찰 증상군과 대응 처방을 직접 기술
- 육경변증 체계 자체가 귀납적 구조이며, 후대 의가들이 장부론/오행론으로 재해석
- 금원사대가, 청대 온병학파까지 임상에서 효과 없는 사변적 처방은 자연스럽게 도태

**보완 필요한 부분:**
- "사변적 내용의 제거"가 체계적으로 이루어졌다고 보기 어려움 (현대적 임상시험 부재)
- 효과 불분명한 처방도 권위 있는 문헌에 실렸다는 이유로 지속 전승된 경우 존재
- "정제된 귀납적 결과물"보다는 "다양한 임상 전통의 중첩"에 가까움

**총평:** 한의학을 신비화하지 않고 임상 경험 기반의 실용적 의학 전통으로 보는 시각은 건전하며 연구 방향의 출발점으로 적절

### 2.b 현대 의학철학적·임상의학적 관점에서의 평가

**강점:**
- "증상-처방 대응" 구조는 현대 정밀의학(precision medicine)의 바이오마커-치료법 매칭과 구조적으로 유사
- (1)번 접근(기존 처방)과 (2)번 접근(핵심 본초 재조합)을 구분한 것은 실무적으로 중요

**주의 필요한 부분:**
- 변증과 처방의 대응이 일대일이 아닌 경우 빈번 → "후보군과 근거의 가중치"로 출력 필요
- 본초 약리학 근거 수준이 성분별로 극도로 불균등 → 근거 수준(level of evidence) 함께 표시 필요
- "새로운 처방을 낸다"는 부분은 의학적·법적으로 가장 민감 → "제안(suggestion)" 수준으로 한정

**총평:** 전체 프레임워크는 논리적으로 건전하고 현대 의학철학적으로도 방어 가능

### 2.c 실현 가능성 평가

**결론: 개인 규모로 충분히 가능**

주요 포인트:
- RAG 기반 시스템은 개인이 구축 가능한 기술적 난이도
- Python 생태계(LangChain, ChromaDB, Streamlit)의 성숙도가 높아 인프라 부담 적음
- **가장 큰 병목은 코딩이 아니라 데이터 구축** (전체 시간의 60~70%)
- 예상 비용: 연간 $300~$1,000 수준

> 이 단계에서 프로젝트 계획서를 docx 파일로 작성하여 제공 (km_llm_project_plan.docx)

## 3. 핵심 전환: LLM 추론 → 벡터 연산 중심으로

### 3.1 연구자의 통찰

> "자연어로 편하게 사용한다기보다는, 사실 LLM의 작동 원리 자체에 닮아있다고 생각하거든. 각 본초의 양을 벡터화하고, 의서에 있는 증상들을 또 벡터화해서 서로 간의 유사도를 찾게 한다면 어떨까?"

핵심:
- 낮은 단계의 연산(벡터 유사도)으로도 충분
- 실시간 LLM 연결 없이 어느 정도 작동하는 모델이 가능
- LLM은 복합적 추론이 필요할 때만 호출

### 3.2 Claude의 분석: 처방의 벡터적 본질

**처방 = 본초 공간에서의 벡터**

예시 - 마황탕: "마황 9g, 계지 6g, 행인 6g, 감초 3g"
→ 본초 300종을 차원으로 잡으면 대부분이 0이고 4개 차원에만 값이 있는 sparse vector

계지탕과의 코사인 유사도 = 처방 간 유사성의 정량적 표현

**증상도 마찬가지**

변증/증상을 가중치 벡터로 표현 → 환자의 증상 벡터와 각 처방의 적응증 벡터 사이의 유사도 = 처방 추천

### 3.3 3층 아키텍처 제안

| 층 | 설명 | LLM 필요 여부 |
|---|---|---|
| **1층: 벡터 연산** | 처방-본초 행렬(F), 처방-증상 행렬(S), 코사인 유사도 | ❌ 오프라인 |
| **2층: 의미적 임베딩** | 본초 약리 텍스트를 임베딩하여 약리학적 유사성 보완 | △ 구축 시 1회 호출 |
| **3층: LLM 추론** | 복합 증상 해석, 방제학적 평가, 자연어 설명 | ⭕ 선택적 온라인 |

**MVP는 1층만으로 충분히 의미 있는 도구가 됨**

### 3.4 용량 정규화 방식

- **벡터 값 = 해당 본초의 용량 / 처방 총량 (비율)**
- 예: 마황탕에서 마황 = 9/24 = 0.375
- 비율이 절대량보다 처방의 성격을 더 잘 드러냄
- 절대 용량 정보는 별도로 보존

## 4. 기술 스택 결정

### 4.1 확정된 기술 스택

| 영역 | 도구 | 용도 |
|---|---|---|
| 데이터 저장 | JSON → SQLite | 처방/본초/변증 구조화 데이터 |
| 행렬 연산 | NumPy, scikit-learn | 코사인 유사도, 클러스터링, 차원 축소 |
| 데이터 조작 | pandas | DataFrame으로 행렬 빌드 |
| 시각화 | matplotlib, plotly | 처방 공간 시각화, 클러스터 확인 |
| 개발 환경 | Jupyter Notebook | 데이터 탐색, 알고리즘 검증 |
| UI (Phase 1+) | Streamlit | 사용자 인터페이스 |
| 임베딩 (2층) | Claude/OpenAI API (1회) | 약리 텍스트 → dense vector |

### 4.2 작업 환경: Claude Code 권장

**이유:**
- 여러 파일로 구성된 프로젝트를 지속적으로 개발하는 데는 웹 인터페이스의 구조적 한계 (파일 시스템 세션마다 리셋)
- Claude Code에서는 로컬 파일 시스템에서 직접 작업, Git 버전 관리 가능
- Jupyter Notebook과 연동하여 데이터 탐색/시각화 병행에 유리

**연구자 프로그래밍 수준: Python 환경 있지만 초보**
→ 오히려 Claude Code가 더 편할 수 있음 (코드 작성은 Claude가, 데이터와 방향성은 연구자가)

## 5. 데이터 설계 결정 사항

### 5.1 연구자 응답 기반 결정

| 항목 | 결정 |
|---|---|
| 데이터 소스 | 교과서/원전(PDF), 일부 엑셀/DB, 머릿속 임상 지식 + 문헌 참조 |
| 시드 데이터 계열 | **상한론 경방** (마황탕, 계지탕, 소시호탕 등) |
| 약리학적 데이터 수준 | **추후 addon** (MVP에서는 성미귀경 + 효능 수준) |

### 5.2 데이터 스키마 요약

**formulas.json (처방)**
- formula_id, name_kr/cn, source, composition(herb_id, role, dose_g), indications(syndromes, symptoms, tongue, pulse), contraindications
- 추후 확장: pharmacology(mechanisms, evidence_level, references)

**herbs.json (본초)**
- herb_id, name_kr/cn/latin, category, properties(nature, flavors, meridians), functions, typical_dose, interactions
- 추후 확장: pharmacology(active_compounds, actions, pathways)

**syndromes.json (변증)**
- syndrome_id, name_kr/cn, system, symptoms(required + optional, 각각 weight 포함), primary/secondary_formulas, differential

### 5.3 벡터화 규칙

**Formula-Herb 행렬 (F):**
- 행: 처방, 열: 본초, 값: dose_g / total_dose_g (비율)

**Formula-Symptom 행렬 (S):**
- 행: 처방, 열: 증상 (전체 합집합), 값: 가중치
- primary_formula의 required 증상: weight × 1.0
- primary_formula의 optional 증상: weight × 0.5
- secondary_formula인 경우: weight × 0.3

**환자 증상 벡터 (q):**
- 동일 증상 차원, 호소 증상: 1.0 (심도에 따라 0.5~1.5)

## 7. MVP 알고리즘 파이프라인

```
[환자 증상 입력]
    ↓
[증상 벡터 q 생성] — 증상 차원에 가중치 배치
    ↓
[cos(q, S[i])] — S 행렬의 각 처방 적응증 벡터와 유사도 계산
    ↓
[유사도 순 정렬] — Top-k 후보 처방 추출
    ↓
[F 행렬 조회] — 후보 처방의 본초 구성, 군신좌사 역할 표시
    ↓
[출력] — 추천 처방 + 유사도 점수 + 본초 분석 + 처방 간 비교
```

**전체 과정이 NumPy 코사인 유사도 연산 — LLM 불필요**

## 9. 개발 로드맵

### Phase 0 (1~2개월): 데이터 + 행렬 구축
1. `src/load_data.py` — JSON → pandas DataFrame
2. `src/build_matrices.py` — F 행렬, S 행렬 생성
3. `notebooks/01_explore.ipynb` — 데이터 탐색, 행렬 시각화, 유사도 테스트

### Phase 1 (1~2개월): 핵심 기능 구현
4. `src/engine.py` — 코사인 유사도 기반 추천/분석 엔진
   - `recommend(symptoms)` → top-k 처방 + 유사도 점수
   - `decompose(formula_id)` → 본초별 역할, 비중 분석
   - `compare(formula_id_a, formula_id_b)` → 유사도, 공통/차이 본초
5. `src/app.py` — Streamlit UI

### Phase 2 (2~3개월): 확장
- 데이터 확장 (처방 300개, 본초 100종)
- 2층 의미적 임베딩 추가
- 환각 방지 전략 구현

### Phase 3 (2~3개월): 검증
- 임상 전문가 평가
- 정확도 벤치마크
- 논문 작성

## 10. 핵심 주의사항

1. **시드 데이터 검증 필수**: Claude가 생성한 시드 데이터의 용량 환산, 군신좌사 배정, 증상 가중치 등은 연구자가 직접 검증·수정해야 함
2. **출력은 항상 "참고용"**: 최종 의사결정은 임상의에게 있음을 모든 응답에 포함
3. **근거 수준 투명 표시**: 약리학적 데이터 추가 시 evidence_level 필수
4. **법적·윤리적 경계**: 의료기기 해당 여부 확인, 연구 목적과 임상 활용 경계 명확화
